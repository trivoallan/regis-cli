# Design: Rapport HTML en fichier unique

**Date:** 2026-04-25
**Scope:** Nouveau format de sortie `--format html` générant un fichier `report.html` autonome (HTML + CSS pur, sans JS, sans dépendances externes).

---

## Contexte

Le rapport HTML actuel (`--format html`) génère une SPA Docusaurus multi-fichiers (React + Tremor + Tailwind). Ce format est bien adapté à l'hébergement sur Pages GitLab/GitHub mais inadapté au partage simple (email, Slack, artifact CI téléchargeable, archivage long terme).

## Objectifs

- Un seul fichier `.html` autonome, lisible hors ligne et en pièce jointe.
- HTML + CSS natifs, aucun JavaScript, aucune ressource externe.
- Contenu configurable via `--sections`.
- Rendu Python (Jinja2), aucune dépendance Node.js.

---

## Changements CLI

### Renommage des formats

| Ancien          | Nouveau              | Description                                           |
| --------------- | -------------------- | ----------------------------------------------------- |
| `--format html` | `--format html-site` | SPA Docusaurus multi-fichiers (comportement inchangé) |
| _(nouveau)_     | `--format html`      | Fichier unique `report.html`                          |

### Nouvelle option `--sections`

```sh
--sections all           # (défaut) toutes les sections avec détails
--sections summary       # playbook + une ligne de stats par analyser, sans findings
--sections trivy,hadolint,freshness  # seulement les analyzers listés
```

- Ignorée pour tous les autres formats.
- Valeur par défaut : `all`.
- Les slugs correspondent aux noms des analyzers (`trivy`, `hadolint`, `skopeo`, `sbom`, `freshness`, `size`, `popularity`, `endoflife`, `scorecarddev`, `provenance`, `versioning`, `dockle`).
- Un slug inconnu provoque un avertissement CLI (`click.echo(..., err=True)`) et est ignoré silencieusement (pas d'erreur fatale).

## Architecture

### Nouveaux fichiers

```text
regis/
  report/
    html.py                     # render_html_single(report, sections) -> str
  templates/
    html/
      report.html.j2            # Template Jinja2 principal
```

### Fichiers modifiés

- **`regis/utils/report.py`** — ajout branche `elif fmt == 'html':` → `render_html_single()` + renommage branche `html` → `html-site`
- **`regis/commands/analyze.py`** — ajout option `--sections`, mise à jour choix `--format`

### Flux de rendu

```text
regis analyze --format html --sections all
  → render_and_save_reports(formats=['html'], sections='all')
    → render_html_single(report, sections=['all'])
      → Jinja2 Environment + report.html.j2
      → str (HTML complet, CSS inline)
    → write_report() → report.html
```

`render_html_single` suit le même pattern que `_render_markdown()` : prend un `dict`, retourne une `str`.

## Structure du template HTML

```html
<html>
  <head>
    <meta charset="utf-8" />
    <meta name="viewport" content="width=device-width, initial-scale=1" />
    <title>{{ image_ref }} — regis report</title>
    <style>
      /* reset, typo, tables, badges — ~100 lignes, 0 dépendance externe */
    </style>
  </head>
  <body>
    <!-- 1. Header : image ref, registry, tag, digest, timestamp, snapshot_date -->
    <!-- 2. Playbook summary (conditionnel : si playbook présent dans le rapport) -->
    <!-- 3. Sections analyzer (conditionnelles selon --sections) -->
    <!-- 4. Footer : "Generated by regis vX.Y.Z on <date>" -->
  </body>
</html>
```

### Section analyser (par analyser)

Chaque analyser présent dans `report` génère un bloc `<section>` :

```html
<section id="trivy">
  <h2>trivy</h2>
  <!-- Tableau stats : colonnes clés (ex. total CVEs, critical, high) -->
  <table class="stats">
    ...
  </table>
  <!-- Tableau findings : affiché seulement si sections != summary -->
  <table class="findings">
    ...
  </table>
</section>
```

En mode `summary` : seul le tableau `stats` est rendu.
En mode `all` ou slug listé : les deux tableaux sont rendus.

### Badges de sévérité

Rendus en CSS pur : `<span class="badge badge-critical">CRITICAL</span>`.

### Contrainte absolute : zéro ressource externe

Ni Google Fonts, ni CDN CSS/JS. Le fichier fonctionne hors ligne et en pièce jointe email.

  - rendu avec rapport minimal (request seulement)
  - rendu avec playbook
  - sections `summary` vs `all` vs liste de slugs
  - vérification absence de ressources externes dans le HTML généré
